```markdown
# Prompt: Qobuz Token Auto-Relogin Detection

## Goal
Make `tools/auth qobuz` (and `ts-auth`) detect an expired Qobuz `user_auth_token`
and surface a loud, actionable error instead of silently continuing with stale creds.

## Problem
`tools/auth qobuz` only refreshes app credentials (`app_id`/`app_secret`).
The `user_auth_token` has no auto-refresh path — Qobuz has no refresh-token mechanism.
When the session expires, `ts-enrich` returns `Genre: 0` with no warning.
The operator has no signal that re-login is needed.

## Required changes

### 1. `tagslut/providers/qobuz.py`
- After loading `user_auth_token` from `tokens.json`, make a lightweight probe:
  `GET https://www.qobuz.com/api.json/0.2/user/get?user_id=<uid>`
  with `X-User-Auth-Token` header.
- If response is 401 or 400, raise `QobuzTokenExpiredError` with message:
  ```
  Qobuz user_auth_token expired. Re-login with:
    cd ~/Projects/tagslut && poetry run python -m tagslut auth login qobuz --email EMAIL --force
  ```

### 2. `tools/auth` (qobuz branch)
- After syncing app credentials, call the same probe.
- If token is dead, print the re-login instruction in red and exit 1.
- If token is valid, print `Qobuz user_auth_token valid ✓`.

### 3. `ts-enrich` startup
- Before beginning enrichment, call `tools/auth qobuz --check-only`.
- If it exits non-zero, abort with the re-login instruction. Do not start enriching.

## Tests
- `tests/providers/test_qobuz_auth.py`: mock a 401 response and assert `QobuzTokenExpiredError` is raised.
- `tests/tools/test_auth_qobuz.py`: mock a 401 and assert exit code 1 and message contains `auth login qobuz`.

## Acceptance criteria
- `ts-auth` with an expired token prints a clear re-login instruction and exits 1.
- `ts-enrich` with an expired token aborts before processing any files.
- All new tests pass.
```

```markdown
# Prompt: --provider Flag for ts-enrich

## Goal
Add `--provider tidal|beatport|qobuz` to `tagslut index enrich` and `tools/enrich`
so the operator can run a targeted single-provider pass.

## Problem
`ts-enrich` hits all three providers for every eligible file on every run.
There is no way to say "run only Qobuz against the 260 genre-missing tracks".
This wastes TIDAL and Beatport rate-limit budget and makes targeted recovery impossible.

## Required changes

### 1. `tagslut/commands/enrich.py`
- Add `--provider` option: `Optional[Literal["tidal", "beatport", "qobuz"]]`.
- When set, skip all other providers in the enrichment loop.
- When set, default the eligibility filter to tracks missing the fields that provider
  supplies: `qobuz → missing genre/label`, `tidal → missing bpm/key`,
  `beatport → missing bpm/key/genre`.

### 2. `tools/enrich`
- Pass `--provider` through to `tagslut index enrich`.

### 3. `ts-enrich` wrapper
- Accept optional positional or flag: `ts-enrich qobuz` → passes `--provider qobuz`.

## Tests
- `tests/commands/test_enrich_provider_filter.py`:
  - Assert that with `--provider tidal`, Qobuz and Beatport clients are never called.
  - Assert that with `--provider qobuz`, only tracks with `canonical_genre IS NULL`
    are included in the eligibility set.

## Acceptance criteria
- `ts-enrich qobuz` processes only tracks with missing genre/label and only calls Qobuz.
- `ts-enrich tidal` processes only tracks with missing bpm/key and only calls TIDAL.
- Existing `ts-enrich` (no flag) behaviour is unchanged.
- All new and existing tests pass.
```

```markdown
# Prompt: Streamrip Tag Passthrough at Registration

## Goal
Extend `tagslut index register --source qobuz` to read Vorbis tags already embedded
by streamrip in downloaded FLACs and write them directly to `track_identity` canonical
columns — without queuing the track for API enrichment.

## Problem
Files at `/Volumes/MUSIC/mdldev/StreamripDownloads` already have genre, label, BPM,
key, and ISRC embedded by streamrip. The current register flow ignores these tags
and queues every file for API enrichment, wasting quota and rate-limit budget.

## Required changes

### 1. `tagslut/ingestion/register.py`
- After building the `TrackIdentity` row, attempt to read Vorbis comment tags
  from the FLAC using `mutagen.flac.FLAC`.
- Map tags to canonical columns:
  - `GENRE` → `canonical_genre`
  - `ORGANIZATION` or `LABEL` → `canonical_label`
  - `BPM` or `TBPM` → `canonical_bpm`
  - `KEY` or `INITIALKEY` → `canonical_key`
  - `ISRC` → `isrc`
- If all four canonical fields are populated from file tags, set `enriched_at = NOW()`
  and skip API enrichment queue.
- If only some fields are populated, write what's available and queue for targeted
  enrichment only for the missing fields.

### 2. `tagslut/commands/register.py`
- Add `--read-tags-from-files` flag (default: True when `--source qobuz`).

### 3. Logging
- For each file where tags are read, log at INFO:
  `[tag-passthrough] Artist - Title: genre=X label=Y bpm=Z key=W`

## Tests
- `tests/ingestion/test_register_tag_passthrough.py`:
  - Mock a FLAC with full Vorbis tags → assert `enriched_at` is set and no API call made.
  - Mock a FLAC with partial tags (genre only) → assert genre written, `enriched_at` NOT set.
  - Mock a FLAC with no tags → assert standard registration flow runs.

## Acceptance criteria
- `ts-get <qobuz_url>` + register writes genre/label to DB from file tags.
- No API calls made for fields already present in file tags.
- All new tests pass.
```
